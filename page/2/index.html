同源策略与跨域

发表于 2019-01-16 | 分类于 web前端

###同源策略

同源策略：是一种处于浏览器安全考虑的一种策略，同源策略只允许本域下的接口交互，不同源的客户端脚本在没有明确授权情况下，不能读写对方资源

“协议+域名+端口“三者相同 才是同源

那么是出于什么安全考虑才会引入这种机制呢？

其实主要是用来防止 CSRF 攻击的

CSRF(Cross-site request forgery), 即跨站请求伪造，指的是黑客诱导用户点击链接，打开黑客的网站，然后黑客利用用户目前的登录状态发起跨站请求

如何预防CSRF攻击：

1.利用Cookie的SameSite属性

在 Cookie 有一个叫SameSite的字段，

可以将SameSite设置为Strict，在Strict模式下，浏览器完全禁止第三方请求携带Cookie

2.使用TOKEN：

首先，浏览器向服务器发送请求时，服务器生成一个字符串，将其植入到返回的页面中。

然后浏览器如果要发送请求，就必须带上这个字符串，然后服务器来验证是否合法，如果不合法则不予响应。这个字符串也就是CSRF Token，通常第三方站点无法拿到这个 token, 因此也就是被服务器给拒绝

跳过CSRF，我们重新回到同源策略，怎样才能向不同域名发送AJAX请求呢？

这时候就得用到跨域了

跨域：由于同源策略的影响，当我们从一个域名的网页去请求另一个域名的资源时，就无法成功获取资源。如果我们要想成功获取资源，那么就要用到跨域

解决跨域问题

####JSONP

JSONP之所以能够用来解决跨域方案,主要是因为 <script> 脚本拥有跨域能力

实现方法：构建一个script标签，它的src属性内容为接口地址并提供一个回调函数

然后接口一般会将返回的数据用函数进行包裹，浏览器端的script标签会直接运行，所以为运行对应的函数拿到返回的数据

但JSONP只限于get请求，基本上被淘汰使用了。

####CORS(跨域资源共享)

CORS(Cross-origin resource sharing)用于让网页的受限资源能够被其他域名的页面访问的一种机制。

通过该机制，页面能够自由地使用不同源的图片、样式、脚本、iframes以及视频。一些跨域的请求（特别是Ajax）常常会被同源策略所禁止的。跨源资源共享定义了一种方式，为的是浏览器和服务器之间能互相确认是否足够安全以至于能使用跨源请求

CORS 需要浏览器和后端同时支持。

浏览器会自动进行 CORS 通信，实现 CORS 通信的关键是后端。只要后端实现了 CORS，就实现了跨域。

服务端设置 Access-Control-Allow-Origin 就可以开启 CORS。 该属性表示哪些域名可以访问资源，如果设置通配符则表示所有网站都可以访问资源

####document.domain

该方式只能用于二级域名相同的情况下，比如 a.test.com 和 b.test.com 适用于该方式。

只需要给页面添加 document.domain = 'test.com' 表示二级域名都相同就可以实现跨域

Ajax、Fetch以及Axios

发表于 2019-01-15 | 分类于 web前端

什么是Ajax

AJAX（Asynchronous JavaScript and XML），指的是通过 JavaScript 的异步通信，从服务器获取 XML 文档从中提取数据，再更新当前网页的对应部分，而不用刷新整个网页

简单的来说Ajax就是用JS向服务端发送请求，然后可以得到服务器响应的内容

原生Ajax

AJAX技术的核心环节：XMLHttpRequest对象

XMLHttpRequest对象是浏览器提供的一个API，用来顺畅地向服务器发送请求并解析服务器响应，当然整个过程中，浏览器页面不会被刷新

实现步骤：

  1. 创建XMLHttpRequest实例
  2. 打开请求: 发送请求之前的一些配置项
  3. 事件监听：一般监听的都是onreadystatechange事件
  4. 发送请求

readyState 代理当前所处的状态：
0 代理被创建，但未调用 open()方法
1 open() 被调用，send() 未调用
2 send() 被调用，且头部和状态已经可获得
3 响应体下载中；responseText 中已获取部分数据
4 下载操作已完成

jQuery中Ajax

JQuery的Ajax是对原生XHR的封装，并且还支持JSONP，非常的方便，用的人也特别多，不过随着Vue，React等框架的流行，如果我们想使用JQuery的Ajax，就得引入整个jQuery，未压缩的有266kb，压缩以后也有86kb，显然有点小题大做，非常不合理，于是就有了Fetch

Fetch

fetch号称是ajax的替代品，它的API是基于Promise设计的

fetch是比较底层的API，相对于jQuery Ajax来说,很多情况下都需要我们再次封装，在使用上自然没有他们方便，而且还有一个写缺点

Fetch的缺点：

  • 只对网络请求报错，对400，500都当作成功的请求
  • Fetch 请求默认是不带 cookie 的，需要设置 fetch(url, {credentials: 'include'})

下一代的HTTP库：Axios

Axios本质上也是对原生XHR的封装，只不过它是Promise的实现版本，并且Vue作者尤雨溪推荐大家用axios替换JQuery ajax

它有以下几大特性：

  • 可以在node.js中使用
  • 提供了并发请求的接口
  • 支持Promise API

GET：

POST

Axios既提供了的封装，也没有之前提到的fetch的各种问题，而且体积也较小，现在我们大可以放心使用Axios
